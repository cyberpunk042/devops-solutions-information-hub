# Gateway identity parser fragility + forwarder contribute-target bug (2026-04-17)

## Summary

TWO-PART FINDING from OpenFleet solo-session 2026-04-17.

PART 1 — Identity parser fragility (brain side):
tools/gateway.py query_identity() at line 851-874 does literal substring match for 'Identity Profile' in a CLAUDE.md heading line. OpenFleet's CLAUDE.md used '## Project Identity (Goldilocks — stable fields only)' — semantically identical, parser silently fails. 'PROJECT IDENTITY: not fully configured' was emitted despite a valid table being present below.

Reproduction:
  1. Project CLAUDE.md uses any heading variant: 'Project Identity', 'Project Profile', 'Goldilocks Identity', etc.
  2. Run gateway status
  3. Parser misses the table → 'not fully configured' emitted

Suggested fix: relax to a regex like:
  re.search(r'\b(Identity.*Profile|Project.*Identity|Identity.*(?=\s*\())', line, re.IGNORECASE)

Alternative: codify the heading format in wiki-schema.yaml so consumers have a hard spec. Per 'Declarations Aspirational Until Infrastructure Verifies', the brain's prescribed template is currently aspirational — no gate verifies the consumer used the right heading string.

OpenFleet-side resolution (already applied 2026-04-17): heading renamed to '## Identity Profile (Goldilocks — stable fields only)' to match brain's template. Parser now detects us correctly. But lingers as a fragility for future consumers.

PART 2 — Forwarder bug on contribute (openfleet side, affects brain intake):
OpenFleet's tools/gateway.py is a forwarder generated by brain's setup.py --connect. It auto-adds '--wiki-root /home/jfortin/openfleet' to every invocation. For commands like lint, validate, status — this is correct (operate on our wiki). For 'contribute' — this is wrong. op_contribute() in brain's gateway at line 1399 writes to paths['wiki'] which resolves from --wiki-root. So 'python3 -m tools.gateway contribute ...' from OpenFleet lands contributions in /home/jfortin/openfleet/wiki/log/ instead of the brain's log/.

Impact: any 'contribution' filed via our forwarder since it was created has landed in our own wiki, not in the brain's intake. Discovered today after two correction/remark contributions landed in openfleet/wiki/log/ instead of brain/wiki/log/.

Root cause: contribute's natural target is the brain (that's the whole point of contribute). But the forwarder was written for lint-style operations where the project wiki IS the target.

Suggested fixes:
  (a) Brain's op_contribute should default to paths['brain'] not paths['wiki'], so --wiki-root gets ignored for contribute (intent: contribute TO brain even when running FROM project wiki).
  (b) OR: forwarder at consumer projects should detect the 'contribute' subcommand and omit --wiki-root for that one.
  (c) OR: add a --target flag to contribute with values brain/local/self, default brain.

Status: OpenFleet re-filed its 2 recent contributions directly against brain by invoking brain's gateway without --wiki-root. The 2 misfiled copies in openfleet/wiki/log/ will be removed. This is consumer-side cleanup; the brain-side bug remains.
