# E012 — Custom Model Library (Unsloth LoRAs on Wiki Corpus)

## Summary

Build a library of small, fine-tuned, domain-specialized models for the second brain's ongoing operations: **Wiki-Assistant** (methodology-fluent chat assistant, Candidate A), **Wiki-Router** (task classifier for AICP, Candidate D), and a **multi-LoRA adapter architecture** (one base + many adapters, Candidate E). With Kimi K2.6 now dominant for frontier capability (E007/E008), the custom-model strategy refocuses on what K2.6 inherently lacks: **wiki-domain specialization** (the 4 principles, 16 models, 22 standards, 10 verbs), **fast local routing classification**, and **privacy-locked offline capability** for specific task classes. Uses Unsloth for training (LoRA rank 32-64, Qwen3.5-4B base, ~2-6 hours on consumer hardware). Not critical for the 2026-04-27 deadline — this is the post-milestone evolution of the stack.

## Operator Directive

> "Lets also start exploring the custom models / custom training I could do such as a model for the information hub / second-brain for example."

> "I don't want to have to deal with Anthropic and Claude and Opus in the future."

## Goals

- Unsloth toolchain installed on operator's workstation (GPU build, 19 GB VRAM sufficient).
- **Wiki-Assistant (Candidate A)** trained — Qwen3.5-4B + LoRA rank 32, 3 epochs, on ~200-500 SFT pairs generated by Unsloth Data Recipe from the wiki corpus.
- **Wiki-Router (Candidate D)** trained — Qwen3.5-0.5B or 1.5B, task-class classifier for AICP's complexity scorer.
- **Multi-LoRA adapter architecture (Candidate E)** scaffolded — one base + 3-5 adapters swapped per task (methodology-assistant, compliance-checker, relationship-suggester).
- Each model has an evaluation holdout and a documented accuracy baseline.
- Integration with AICP routing (E011): which local tier picks up which custom-trained model for which task class.
