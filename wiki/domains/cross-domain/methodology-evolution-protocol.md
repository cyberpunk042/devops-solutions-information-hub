# Methodology Evolution Protocol

## Summary

How the methodology evolves based on evidence, not theory. The research wiki owns the canonical methodology; consumer projects provide evidence (violation rates, agent concerns, compliance reports). Evolution follows a cycle: collect evidence → identify patterns → propose changes → test in wiki → propagate to consumers. The agent follows the current version exactly — it does NOT evolve the methodology. The operator does.

## Key Insights

1. **Evolution is evidence-driven, not theory-driven** — methodology changes must trace to specific failures, violation patterns, or compliance data. "I think this would be better" is not sufficient. "Agents violated stage boundaries 75% of the time in overnight runs" IS sufficient.

2. **The wiki is the laboratory, projects are the field** — changes are tested in the research wiki first (where the cost of failure is low), then propagated to consumer projects once proven.

3. **Versioned and backward-compatible** — methodology.yaml carries a version. Consumer projects pin to a version and upgrade explicitly. Breaking changes require a migration guide.

4. **The agent never evolves the methodology** — agents follow the current version. The operator reviews evidence and makes changes. This prevents runaway self-modification where agents weaken their own constraints.

## Deep Analysis

### The Evolution Cycle

> [!abstract] Five-Step Evolution Cycle
>
> | Step | Actor | Input | Output |
> |------|-------|-------|--------|
> | 1. Collect | Automated + operator | Session logs, compliance reports, agent concerns | Evidence corpus |
> | 2. Identify | Operator | Evidence corpus | Patterns (repeated violations, false positives, gaps) |
> | 3. Propose | Operator + wiki | Patterns | Proposed changes with rationale |
> | 4. Test | Wiki agents | Proposed changes | Results in wiki environment |
> | 5. Propagate | Operator | Validated changes | Updated methodology.yaml + migration guide for consumers |

### Evidence Sources

> [!info] Where Evolution Evidence Comes From
>
> | Source | What It Shows | How to Collect |
> |--------|--------------|----------------|
> | Completion logs | Which stages were executed, which artifacts produced | Auto-generated by harness (Tier 4) or manual logging |
> | Agent concerns | What the agent flagged as wrong (design mismatch, potential bug, scope wrong) | `/concern` command or concern sections in completion logs |
> | Compliance reports | Violation rates, false positives, common failure modes | Post-session analysis of git history vs methodology expectations |
> | Session timelines | How long each stage took, where agents got stuck | Wiki log entries (wiki/log/) |
> | Operator overrides | When the operator told the agent to ignore methodology | Directive logs in raw/notes/ |

### Version Management

```yaml
# In methodology.yaml header
# Version: 1.0 (2026-04-11)
# Changes from: initial release
# Breaking: none (first version)
```

**Versioning rules:**
- **Patch** (1.0 → 1.0.1): Fix typos, clarify wording, add examples. No behavioral change.
- **Minor** (1.0 → 1.1): Add new model, add optional artifacts, extend domain profiles. Backward compatible.
- **Major** (1.0 → 2.0): Change stage sequences, remove models, change required artifacts. Breaking — requires migration guide.

**Consumer project pinning:**

```yaml
# In consumer project's methodology-overrides.yaml
methodology_version: "1.0"  # pin to specific version
auto_upgrade: minor          # auto-accept minor, require manual for major
```

### Propagation Mechanism

> [!tip] How Changes Reach Consumer Projects
>
> | Mechanism | When | How |
> |-----------|------|-----|
> | Git submodule | Consumer pins wiki as submodule, pulls updates | `git submodule update --remote` |
> | File copy | Consumer copies config files manually | Compare versions, copy updated files |
> | Export tool | Wiki's export pipeline includes methodology config | `python3 -m tools.export methodology` |
> | Notification | Wiki generates changelog when methodology changes | Wiki log entry + tag in manifest.json |

The export tool approach is preferred — it packages methodology.yaml + domain profiles + artifact-types.yaml into a consumable bundle.

### What Triggers Evolution

> [!warning] Evolution Triggers — When to Update the Methodology
>
> | Trigger | Signal | Action |
> |---------|--------|--------|
> | High violation rate | >50% of agents violate a specific rule | Investigate: is the rule unclear, too strict, or wrong? |
> | False positives | Validation flags correct work as violations | Adjust thresholds or add exceptions |
> | Missing model | Work doesn't fit any existing model | Define new model with stages and chain |
> | Stage boundary too rigid | Legitimate work blocked by forbidden_outputs | Consider: is the boundary wrong, or is the agent using the wrong model? |
> | New domain | Project with a stack not covered by existing profiles | Create new domain profile |
> | Agent fatigue patterns | Quality cliff at consistent point | Adjust session limits or task grouping |
> | Operator overrides cluster | Operator repeatedly overrides the same rule | The rule may be wrong |

### What Does NOT Trigger Evolution

> [!warning] Not Grounds for Methodology Change
>
> - A single agent session going wrong (one data point is not a pattern)
> - Agent requesting methodology relaxation (agents optimize for completion, not quality)
> - "It would be faster if..." (speed is not a goal — quality is)
> - Theoretical improvements without evidence (test in wiki first)

### Recording Changes

Every methodology change gets a wiki log entry:

```markdown
# wiki/log/YYYY-MM-DD-methodology-evolution-vN.md
---
title: "Methodology Evolution vN"
type: note
note_type: session
---

## What Changed
- [specific change 1]
- [specific change 2]

## Evidence
- [link to completion log / compliance report / agent concern]

## Rationale
- [why this change was made based on evidence]

## Impact
- [which projects need to update]
- [migration steps if breaking]
```
