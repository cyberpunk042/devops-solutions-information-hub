## Root Problems Identified

1. **Language bias** — "TypeScript/Node Domain" conflates a language with a way of working. The chain should be about the DOMAIN (e.g., backend API, frontend SPA, CLI tool) not the language.
2. **Project conflation** — OpenArms-specific artifacts are presented as universal TypeScript patterns. The second brain should separate PROJECT KNOWLEDGE from DOMAIN KNOWLEDGE.
3. **Rigidity** — One fixed chain per language. Should show OPTIONS and let the consumer pick what's appropriate for THEIR context (Goldilocks).
4. **Thin content** — Good start but needs depth, alternatives, and proper framing.
5. **Global problem** — This applies to ALL domain chains, not just TypeScript.

## Solution Direction

- Create wiki/projects/ folder for project-specific knowledge (OpenArms patterns, OpenFleet patterns, etc.)
- Refactor domain chains to be about WORK DOMAINS (API backend, frontend, CLI, infrastructure, knowledge) not LANGUAGES
- Each chain should present OPTIONS not a single fixed pipeline
- Chains should reference project examples ("OpenArms uses X for this stage") not embed them as universal truth
- Follow Goldilocks: the chain adapts to context, it doesn't prescribe one path
