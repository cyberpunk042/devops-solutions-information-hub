# User Directive — 2026-04-09 — Methodology Standards & Research-First

## Verbatim

> we need to be able to define methodologies and group and order and cases and the documents and artifacts and what they must contain and in what order and using which way.
> Like this requirements spec + artifactA, B, C, d,....
> Its not set in stone its per-case, per-size, per-type of task or project or epic or module....
> The more we are clear and we have a standard we can share the better and the more we can use the norms mantality like the Skills and Design and other things like LLM WIKI and so on, the better. research is key, on our projects and online.
>
> Then we can start brainstoming... and again.. I told you it require a lot of research....

## Interpretation

### What the user wants
A methodology framework that defines:
- What documents/artifacts are needed for each type of work (project, epic, module, task, feature)
- What each artifact MUST contain
- In what ORDER to produce them
- Using which APPROACH (brainstorm? spec? design? scaffold?)
- Per-case flexibility: not one-size-fits-all, but typed templates per scale/type

### The approach must be RESEARCH-FIRST
- Research existing standards FIRST (SDLC, Agile, TOGAF, C4, ADR, etc.)
- Research what the ecosystem projects already do (openfleet, AICP, OpenArms, control-plane)
- Research online patterns (Skills, Design.md, AGENTS.md, LLM Wiki, etc.)
- THEN brainstorm with the user
- Don't jump to design without research

### This is NOT just about the wiki
This is about defining how ALL work gets done across the ecosystem. The methodology becomes a transferable standard shared between projects.

### Action
1. Deep research on software development methodologies + artifact standards
2. Deep research on the ecosystem's existing patterns
3. Research the emerging LLM-agent methodology patterns (superpowers, spec-driven dev, etc.)
4. THEN brainstorm the methodology framework
5. THEN design the wiki structures that support it
