# Sync Service IaC — Design Spec

## Problem

The wiki sync and watcher daemons exist as Python tools (`tools/sync.py --watch`, `tools/watcher.py --watch`) but there's no reproducible way to deploy them as persistent services. The previous attempt to manually create a systemd unit file was rightly rejected — infrastructure must be generated by tooling, not hand-crafted.

The user works in WSL2 and needs the sync daemon running persistently so Obsidian on Windows always sees the current wiki state. Without this, every session requires a manual `python -m tools.sync` which defeats the purpose of having a sync tool.

## Solution

Extend `tools/setup.py` with a `--services` subcommand that generates systemd user unit files from templates in `config/services/`, installs them to `~/.config/systemd/user/`, and enables them via `systemctl --user`.

## Architecture

```
config/services/
├── wiki-sync.service.template       # Sync daemon template
└── wiki-watcher.service.template    # Watcher daemon template

tools/setup.py --services [name]
├── No name → list available templates
├── wiki-sync → generate + enable + start sync daemon
└── wiki-watcher → generate + enable + start watcher daemon
```

Templates use `{{placeholder}}` variables filled at install time. Services are independent and opt-in — the user chooses which to enable.

## Service Templates

### wiki-sync.service.template

```ini
[Unit]
Description=Research Wiki Sync (WSL → Windows)

[Service]
Type=simple
WorkingDirectory={{project_root}}
ExecStart={{venv_python}} -m tools.sync --watch
Restart=on-failure
RestartSec=10
Environment=PYTHONPATH={{project_root}}

[Install]
WantedBy=default.target
```

### wiki-watcher.service.template

```ini
[Unit]
Description=Research Wiki Watcher (change detection + post-chain)

[Service]
Type=simple
WorkingDirectory={{project_root}}
ExecStart={{venv_python}} -m tools.watcher --watch
Restart=on-failure
RestartSec=10
Environment=PYTHONPATH={{project_root}}

[Install]
WantedBy=default.target
```

Note: services are independent. wiki-watcher does NOT auto-trigger sync. If the user wants both, they enable both. Concerns stay separated.

## setup.py Changes

### New function: install_service()

```python
def install_service(service_name: str, project_root: Path) -> bool:
```

1. Read template from `config/services/{service_name}.service.template`
2. If template doesn't exist, list available templates and return False
3. Fill placeholders: `{{project_root}}`, `{{venv_python}}`
4. Write to `~/.config/systemd/user/{service_name}.service`
5. Run `systemctl --user daemon-reload`
6. Run `systemctl --user enable {service_name}`
7. Run `systemctl --user start {service_name}`
8. Print status info (unit path, status command, log command)

### New function: list_services()

```python
def list_services(project_root: Path):
```

1. Glob `config/services/*.service.template`
2. For each, extract service name and check if already installed (unit file exists in systemd dir)
3. Print table: name, installed status, running status (via `systemctl --user is-active`)

### CLI changes

Add `--services` argument:
- `setup.py --services` → call `list_services()`
- `setup.py --services wiki-sync` → call `install_service("wiki-sync", ...)`

### Platform gate

If not WSL/Linux (detected via `is_wsl()` or `platform.system() == "Linux"`), print "systemd services only supported on Linux/WSL" and exit. No silent failure.

## CLAUDE.md Update

Add to the Setup section:

```
python -m tools.setup --services              # List available services
python -m tools.setup --services wiki-sync    # Deploy sync daemon (WSL→Windows)
python -m tools.setup --services wiki-watcher # Deploy watcher daemon (auto post-chain)
```

## Runtime Management

Setup generates and enables. Runtime management uses systemctl directly:

```
systemctl --user status wiki-sync
systemctl --user stop wiki-sync
systemctl --user start wiki-sync
systemctl --user disable wiki-sync
journalctl --user -u wiki-sync -f
```

## Success Criteria

- `setup.py --services` lists wiki-sync and wiki-watcher with installed/running status
- `setup.py --services wiki-sync` generates unit file, enables, starts the service
- `systemctl --user status wiki-sync` shows active (running)
- Killing the terminal does NOT stop the sync daemon
- A fresh clone can deploy services by running `setup.py --deps && setup.py --services wiki-sync`
- Template files are version-controlled in `config/services/`
